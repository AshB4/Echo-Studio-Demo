# DEV.to Style Guide

Purpose: collect examples and notes for future Taste Skill voice passes on DEV.to posts and technical article drafts.

## Examples to Collect

- Technical posts with clear problem, context, implementation, and result.
- Debugging writeups that show the investigation path.
- Architecture notes that explain tradeoffs.
- Tooling posts with concrete commands or examples.
- Strong introductions that avoid generic tutorial openings.
- Posts that generated saves, comments, or meaningful developer discussion.

For each example, capture:

- Date.
- Article title.
- Tags.
- Full draft or canonical URL.
- Code snippets used.
- Result, if known.
- Why it should guide future drafts.

## Writing Characteristics to Preserve

- Concrete technical detail.
- Honest tradeoffs.
- Clear sequence of decisions.
- Plain explanations.
- Developer respect: no over-explaining obvious context.
- Specific failure modes and fixes.

## Formatting Preferences

- Use headings that describe the actual section.
- Prefer code blocks for commands and examples.
- Keep paragraphs short but complete.
- Include prerequisite context before commands.
- Use lists for checklists, failure modes, and steps.
- Keep tags relevant and restrained.

## Platform-Specific Guidance

- DEV readers need substance quickly.
- Do not turn a technical article into a marketing post.
- Preserve canonical URL and attribution expectations when syndicating.
- Include code, config, or process evidence where appropriate.
- Make the introduction specific to the problem, not a generic statement about software.

## Avoid Notes

- Avoid thin "tips" posts with no implementation evidence.
- Avoid vague architecture commentary.
- Avoid tutorial filler.
- Avoid claiming a tool is simple when the setup has real edge cases.
- Avoid publishing without checking code snippets and links.
